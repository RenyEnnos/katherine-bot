**Issue:** Resolves #205

**Causa Raiz:** O backend aceitava passivamente a identidade do usuário a partir do campo `user_id` no payload da requisição ou URL (no caso do histórico). Isso criava uma vulnerabilidade (IDOR) que permitiria a um ator malicioso forjar o campo e acessar dados ou interagir como outra pessoa.

**Solução:**
- A dependência FastAPI `get_current_user` foi introduzida usando `HTTPBearer`. Ela recebe o Bearer token do header e invoca `supabase.auth.get_user(token)` para confirmar a identidade em tempo real junto ao Supabase.
- A rota `POST /chat` remove o envio do `user_id` e deriva isso da validação segura do token.
- A rota `GET /history/{user_id}` garante que o usuário requisitado corresponda ao ID do token que foi validado, rejeitando com HTTP 403.
- No frontend, a autorização com axios injeta ativamente a sessão Supabase na API local, resolvendo a vulnerabilidade de forma alinhada.

**Arquivos Alterados:**
- `backend/main.py`: Lógica de rotas e segurança via dependência.
- `backend/test_auth.py`: Inclusão dos testes mockados de auth requeridos.
- `frontend/src/features/chat/hooks/useChat.js`: Passagem de token.
- `frontend/src/features/chat/services/chatService.js`: Remoção de `user_id` arbitrário no POST.
- `frontend/src/shared/services/apiClient.js`: Injeção via Interceptor do token JWT do Supabase.

**Testes Executados:**
- Criou-se testes mockando a chamada de `supabase.auth.get_user()` para validação completa e unitária: `test_missing_token`, `test_invalid_token`, `test_valid_token` e `test_mismatched_user_id`. (Status: ✅ All passing in pytest)
- Lints e Build do Vite frontend não acusam regressão ou warnings (Status: ✅ Completed)

**Riscos:** Como toda a autorização do sistema mudou de base e requer `HTTPBearer` em requisições antes públicas/inseguras, se a API do Supabase cair, o backend fica paralisado.
**Rollback:** `git revert` na commit PR e voltamos para o estado inicial anterior onde requisições operavam localmente baseando no body do usuário.

**Itens Fora de Escopo:** Não adicionei refatorações na engine nem lidei com cache de perfis conforme AGENTS.md recomenda; não abordei os tickets #206-#211.
